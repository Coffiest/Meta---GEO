# ポーカーロジック仕様メモ(TDAルール準拠 / 実装基礎資料)

このドキュメントは、トーナメントエンジン実装前のルール調査結果を記録したものです。
出典はTDA公式サイト(pokertda.com)、各種ポーカーメディア、Ten Four Poker関連の公開情報(note等)。
**条文番号や一部の細則は複数の二次情報源から再構成したものであり、公式PDF原文で完全一致を確認できていない箇所があります。「不確か」と明記した箇所は実装前に要一次確認。**

## 1. サイドポット計算(レイヤー方式)

各プレイヤーの「そのハンドでの拠出累計額」を基準に、最小拠出額でレイヤーを切り出していく方式で実装する。

```
function buildPots(contributions: Map<Player, int>, foldedPlayers: Set<Player>):
    pots = []
    remaining = copy(contributions)
    while any(remaining[p] > 0 for p in remaining):
        payers = [p for p in remaining if remaining[p] > 0]
        layerAmount = min(remaining[p] for p in payers)
        potAmount = 0
        eligiblePlayers = []
        for p in payers:
            take = min(layerAmount, remaining[p])
            potAmount += take
            remaining[p] -= take
            if p not in foldedPlayers:
                eligiblePlayers.append(p)
        pots.append(Pot(amount=potAmount, eligible=eligiblePlayers))
    return pots  # pots[0]=メインポット, 以降サイドポット
```

- フォールドしたプレイヤーの拠出額はポットに残るが、`eligible` には含めない(取り分の権利を失うだけで、拠出自体は消えない)。
- 清算時: 各ポットごとに `eligible` の中で最強ハンドを判定し、山分け。端数チップはボタンから見て一番近い(反時計回りで最初に座る)勝者に配る(この端数配分ルールはハウスルール寄りで一次確認不十分)。
- 全員オールインでベッティングが早期終了した場合は、残りのアクションを待たず即座に全員のカードをオープンしてからランナウトする(TDA Rule 16相当「Face Up for All-Ins」)。

## 2. 不完全レイズ(インコンプリートレイズ)とベッティングの再オープン ― TDA Rule 47相当

- **ショートコール**(コール額に届かないオールイン): レイズ扱いにならない。後続アクションに制限なし。
- **インコンプリートレイズ**(直前のレイズ幅=increment未満のオールイン):
  - まだこのラウンドでアクションしていないプレイヤーには制限なし。
  - **既にアクション済みのプレイヤーは、フォールドかコールのみ**(再レイズ不可)。
  - 複数の短いオールインが積み重なり、合計incrementが「直近の正当なフルレイズ幅」以上になれば、ベッティングは再オープンされ、既アクション済みプレイヤーも再度レイズ可能になる。再オープン時の最小レイズ幅は「直近の正当なフルレイズ幅」を用いる。

```
function processAllIn(allInAmount, currentBetToMatch, roundLastFullRaiseSize):
    increment = allInAmount - currentBetToMatch
    if increment <= 0: return CALL_SHORT
    if increment >= roundLastFullRaiseSize:
        return FULL_RAISE(reopens=True, newMinRaiseSize=increment)
    else:
        return INCOMPLETE_RAISE(reopensForActedPlayers=False, minRaiseSizeIfReopened=roundLastFullRaiseSize)
```

- 状態として各プレイヤーの「当ラウンドでアクション済みフラグ」と「当ラウンドの直近正当フルレイズ幅」を保持する必要がある。

## 3. ミニマムレイズ

- プリフロップ初回レイズ: 最低 BB の2倍。
- 以降: 直前の正当なレイズ幅以上を上乗せ。
- オールインが最小レイズ額未満 → 上記「インコンプリートレイズ」処理。

## 4. ボタン/ブラインド

- **デッドボタン方式**: BBは必ず本来の順番のプレイヤーが払い、それに応じてSB/ボタンが決まる(空席にSB/ボタンが割り当たることがある)。
- **ヘッズアップ**: デッドボタン適用外。SB=ボタンがプリフロップ最初にアクション、フロップ以降は最後にアクション。

## 5. チップレース(奇数チップ処理)

- 回収対象の低額チップ枚数に応じてカードを引き、ハイカード順(同ランクはスペード>ハート>ダイヤ>クラブ)に上位チップと交換。
- チップレースでプレイヤーの残りチップが0になることは不可(最低額面1枚を保証)。

## 6. タイムバンク/切断時の扱い(業界標準、TDA明文ではなくオンラインルーム慣行)

- 基本アクション時間: 目安10〜30秒(ストリート・大会規定による)。
- タイムバンク: 追加の持ち時間プール。使い切ると自動アクション(チェック可能ならオートチェック、ベットに直面していればオートフォールド)。
- 切断時: オートフォールド継続、ブラインド/アンティは自動徴収され続ける。一定回数の追加時間付与(ディスコネクションプロテクション)を実装するのが一般的。

## 7. ミスディール・露出カード・ストリングベット・角待ち

- ミスディール: 「実質的アクション」発生後は原則宣言不可(デッキ欠陥時を除く)。
- レイズは「ワンモーションでチップを押し出す」または「先に金額を口頭宣言する」のいずれかでのみ有効。曖昧な小分けベット(ストリングベット)は禁止 → UI上はアクション確定を単一操作にすることで自然に担保できる。
- ソフトプレイ・チップダンピング等の共謀行為は失格・チップ没収対象。

## 8. ペナルティ(段階制、条文番号は要一次確認)

1. 口頭警告 → 2. ミスハンド/ミスラウンド(そのテーブル人数×ペナルティラウンド数だけ強制フォールド) → 3. エスカレーション → 4. 失格(チップ没収)。

## 9. Ten Four Poker(参考コンセプト)

- ブラウザベース(PC/スマホ両対応、ネイティブアプリではない可能性)、無料、Fast Fold形式の6-maxリングゲーム(100bbスタート)が中心と見られる。MTT機能の明確な言及は確認できず。
- 「Win Rate(bb/100)」による実力可視化、ハンド履歴の全面公開(相手ハンドも含む)、Expertランク(週1,000ハンド維持要件)が特徴。
- カジュアルさより上達・研究志向のポジショニング。**本アプリは「トーナメント版」として、この思想(ハンド履歴の徹底公開+実力の可視化)を踏襲しつつ、GEO(仮)戦略DBという独自機能を追加する。**

## 10. GTO Wizard UI構成(GEO分析UIの参考)

- Rangesタブ: 両者レンジをサイドバイサイド比較、インクリメンタル検索型フィルタ。
- Solution Browser: ハンドマトリクスにホバーで戦略/EV表示。
- Aggregate Reports: 全フロップ横断の集計、ボードテクスチャ別フィルタ・グルーピング。
- Custom Aggregated Reports: スタック深度・ベットサイズ等を自由設定した独自集計。
- 全体構成: 上部フォーマット選択 → 左右レンジ比較 → 下部フィルタ/レポート、の多層タブ構成。

## 11. 実装上の総括

- サイドポット・不完全レイズ判定はエンジンのコアであり、`packages/engine` として純粋関数・単体テスト可能な形で切り出す。
- 「全員オールイン確定→即オープン→ランナウト」を専用フローとして持つ。
- ラウンドごとの「アクション済みフラグ」「直近正当フルレイズ幅」をベッティングラウンドの状態として保持する設計にする。
